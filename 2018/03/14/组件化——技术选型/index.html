组件化——技术选型

YoungSure 3月 14, 2018
  • component
  • research
  • 分享到微博
  • 分享到 LinkedIn

实现目标

  1. 建立一个可以管理组件的管理平台（于开发者可以查找并直接使用现成的组件，提高开发效率；于产品、设计可以方便查阅已有的基础模块，保持与研发的信息对称，更好的设计、考衡新产品需求设计等；于管理者可以对整个项目的把控更直观全面）；
  2. 抽象、设计、开发出畅途网现有可重复使用的组件（要和产品、设计部门做好充分的沟通，做到统一规范）。

技术预选方案

  1. 基于vue服务端渲染框架nuxt的大方案
  2. 基于vue的组件系统的轻量级方案
  3. 基于freemarker自定义标签的本地方案

基于vue服务端渲染矿建nuxt的大方案：

  1. 技术栈：(vue全家桶 + webapck + nodeJS + express)nuxtJs
  2. nuxt的工作原理如下：
  3. 结合畅途网的组织框图如下：
  4. 可以实现前后端分离，并很好的解决SEO问题（主要就是为了解决这个问题）；
  5. 在不做大改动前端项目（slwt3.0）的前提下，将前端项目作为中间件,nginx分请求类型，将做了SSR的请求代理到nuxt项目上，具体做如何业务处理则由nuxt的服务层（express）和slwt3.0项目做交互，其他请求依然代理到原项目上；
  6. 有比较大的学习成本，对于畅途网来说，vue、webpack、nodejs、express框架都是新技术，同时相对的技术储 备较薄弱，相应的研究、开发、普及周期都会比较长。

基于vue的组件系统的轻量级方案

  1. 技术栈：(vue(component) + requireJs)
  2. 结合畅途网的组织框图如下：
  3. 在对项目的小幅度改动下，可以轻松实现组件化管理（主要依托组件管理平台的建立和设计中心的UI组件的统一实现）；
  4. 学习成本比较小，只需掌握vue组件的相关内容与requireJs的简单应用即可；
  5. 使用这种方案也就意味着舍弃掉对SEO的兼顾，只适合在非SEO要求的页面使用（实际开发中的90%是针对非SEO页面的），对SEO有要求的页面就要使用传统的开发方式（freemarker模板引擎）；
  6. 还有一个缺点就是受限于vue框架本身，兼容性对于IE，只兼容到IE9+。

基于freemarker自定义标签的本地方案

  1. 技术栈：(freemarker 自定义标签)
  2. 要对freemarker的自定义标签做深入研究，使其可以满足组件的基本特征（独立的dom结构、css样式，以及封闭的数据管理、可跨页面复用，最重要的就是要能够对外通信）；
  3. 结合畅途网的组织框图如下：
  4. 这个方案有一个很明显的缺点就是前端页面表示和服务层（java）完全耦合在一起，对于以后的前后端分离甚至日常代码维护和功能拆分等有一定的影响；
  5. 此外，这种方案还会把样式代码、js代码混入到html结构代码中，造成代码混乱，难以调试的问题等。
